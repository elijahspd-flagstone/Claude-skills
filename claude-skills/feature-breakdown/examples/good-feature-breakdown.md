# Example: Good Feature Breakdown

This is a worked example of what excellent output looks like from the
feature-breakdown skill. Use this to calibrate your own PRD quality.

**Input the PM gave:** "Let's add a way for builders to invite teammates to their site projects."

---

````markdown
> PRD generated by Codepup Feature Breakdown
> Feature: Team Collaboration — Project Invites | Date: 2026-03-18 | Status: Draft

---

## Team Collaboration — Project Invites

**Status:** Draft
**Author:** Codepup PM
**Date:** 2026-03-18

### Problem
Builders on Codepup currently work alone. When they grow a team or bring on a
freelancer, there's no way to share access — they end up sharing login credentials,
which creates security risks and makes it impossible to audit who changed what.
This has appeared in 23 support tickets in Q1 and is the top-voted request on our
feedback board.

### Solution
Allow a Builder to invite other users to their project by email. Invitees get a
role-based level of access (Viewer or Editor) and can log in with their own credentials.
The Builder retains full ownership and can revoke access at any time.

### Why Now
We're seeing team plans become 40% of new signups this quarter. Builders on team plans
expect multi-user access — we're losing upgrades to competitors who offer it.

### Success Metrics
| Metric | Baseline | Target | Timeline |
|--------|----------|--------|----------|
| Team plan 30-day retention | 62% | 72% | 60 days post-launch |
| Support tickets re: shared access | 23/month | <5/month | 30 days post-launch |
| Invite-to-accept rate | — | >70% | 30 days post-launch |

### Out of Scope (v1)
- Granular per-page or per-section permissions
- SSO / SAML integration
- Guest access without a Codepup account
- Transferring project ownership

### Open Questions
- [ ] Should Viewer role be available on free plans or only paid? — Owner: Growth — Due: 2026-03-25
- [ ] What happens to a collaborator's access when the owner downgrades? — Owner: Eng — Due: 2026-03-25

---

### User Stories

**Must Have (MVP)**
- As a Builder, I want to invite a teammate by email so that they can help me edit my project without knowing my password.
- As a Builder, I want to assign a role (Viewer or Editor) when inviting so that I control what each person can change.
- As an invitee, I want to receive an email with a clear call to action so that I can accept and start collaborating within minutes.
- As a Builder, I want to see a list of current collaborators so that I know who has access at any time.
- As a Builder, I want to remove a collaborator so that I can revoke access immediately when someone leaves the project.

**Should Have (v1.1)**
- As a Builder, I want to change a collaborator's role after inviting so that I don't need to remove and re-invite them.
- As an Admin, I want to see all collaborator relationships across all projects so that I can audit access for compliance.

**Nice to Have (backlog)**
- As a Builder, I want to invite someone via a shareable link (not just email) so that onboarding is faster.

---

### Edge Cases

**Data & Input**
- [ ] Inviting an email that already has a Codepup account → skip signup, go straight to accept
- [ ] Inviting the same email twice → show "invite already pending" error, don't send a second email
- [ ] Inviting your own email → block with "you already own this project"
- [ ] Invite link clicked after it expires (7 days) → show expiry message with "request a new invite" CTA

**State & Permissions**
- [ ] Free-plan Builder tries to invite when limit is 1 collaborator → paywall modal
- [ ] Invitee accepts but Owner has since deleted the project → "Project no longer exists" screen
- [ ] Editor tries to invite others → "Only the Owner can manage collaborators" — block the action
- [ ] Owner account is suspended → all collaborator access is also suspended

**Async & Timing**
- [ ] Invite email goes to spam → in-app "resend invite" option visible to Owner
- [ ] Owner removes collaborator while collaborator is actively editing → session ends, redirect to "access revoked" screen on next API call
- [ ] Two Editors save conflicting changes at the same time → last-write-wins with a conflict toast notification

**Integrations**
- [ ] Email delivery failure (e.g. SendGrid down) → queue for retry, show "invite is being sent" status
- [ ] Invitee uses social login (Google) with a different email than invited → prompt to confirm identity match

### QA Checklist

**Functional**
- [ ] Owner on a paid plan can invite a new user by email with role Viewer
- [ ] Owner on a paid plan can invite a new user by email with role Editor
- [ ] Invitee receives email within 60s of invite being sent
- [ ] Invite link works and creates a new account if invitee doesn't have one
- [ ] Invite link correctly logs in an existing user who matches the email
- [ ] Owner can see pending and accepted invites in the collaborators list
- [ ] Owner can remove a collaborator — access is revoked immediately
- [ ] Removed collaborator gets 403 on their next API call

**Edge Cases**
- [ ] Duplicate invite shows correct error, no second email sent
- [ ] Self-invite is blocked
- [ ] Expired link shows correct message with resend option
- [ ] Free-plan paywall appears when invite limit is reached

**Cross-browser / Device**
- [ ] Invite panel works on Chrome, Safari, Firefox (latest)
- [ ] Invite email renders correctly in Gmail, Outlook, Apple Mail
- [ ] Invite panel is usable on a 375px mobile viewport

**Accessibility**
- [ ] Email input and role selector are keyboard-navigable
- [ ] Error messages are associated with the relevant input (aria-describedby)
- [ ] "Remove collaborator" confirmation dialog is focusable and dismissable via keyboard

**Analytics**
- [ ] `collaborator_invite_sent` fires once per invite, not per keystroke
- [ ] `collaborator_invite_accepted` fires on the invitee's session, not the owner's
- [ ] `paywall_hit` fires before the invite form is shown to a free user

### Engineering Tasks

**Backend**
- [ ] Create `project_invitations` table with: id, project_id, invitee_email, role, status, token, expires_at, created_at — **M**
  - Acceptance: migration runs cleanly, indexes on project_id and token
- [ ] POST `/projects/:id/invitations` — create invite, send email via SendGrid — **M**
  - Acceptance: returns 201 with invite_id; email received within 60s; duplicate returns 409
- [ ] POST `/invitations/:token/accept` — validates token, creates or links user, adds collaborator record — **M**
  - Acceptance: returns 200 with session token; expired token returns 410
- [ ] DELETE `/projects/:id/collaborators/:user_id` — revoke access — **S**
  - Acceptance: returns 204; subsequent API calls by removed user return 403
- [ ] GET `/projects/:id/collaborators` — list all with role and status — **S**
  - Acceptance: returns array with pending and accepted invites

**Frontend**
- [ ] "Manage collaborators" panel in project settings — invite form + collaborator list — **L**
  - Acceptance: matches Figma spec; empty state, pending state, and accepted state all rendered
- [ ] Invite acceptance flow — landing page for invite link, handles new and existing users — **M**
  - Acceptance: new user lands on signup with email pre-filled; existing user is logged in and redirected to project
- [ ] Paywall modal on invite attempt for free users — **S**
  - Acceptance: modal shows plan comparison; upgrade CTA links to billing page

**Data / Analytics**
- [ ] Instrument all 8 analytics events from the events table — **S**
  - Acceptance: all events appear in Amplitude within 60s in staging

**DevOps / Infra**
- [ ] Add `SENDGRID_INVITE_TEMPLATE_ID` env variable to staging and production — **S**
  - Acceptance: variable documented in .env.example

**QA**
- [ ] Write test scenarios from QA checklist above — **M**
- [ ] Regression test: existing project settings page unaffected — **S**

---

**What to do next**
- Resolve the two open questions (free-plan collaborator limit, downgrade behavior) before the design review
- Growth and Design need to sign off on the paywall modal copy and the invite email before engineering starts
- The riskiest assumption: that a 70% invite acceptance rate is achievable — validate with a manual invite flow to 10 beta users before building the full automation
````

## What Makes This a Good Breakdown

1. **Problem is specific** — it names a source (support tickets, feedback board) and a number (23 tickets). A vague problem produces vague solutions.

2. **Success metrics are measurable** — "team plan retention +10pp" is a real goal; "users will love it" is not.

3. **Out of scope is deliberate** — it names things with a reason. Cuts without reasons get relitigated in sprint planning.

4. **User stories are outcome-focused** — every "so that" states a business or user value, not a technical step.

5. **Edge cases are specific** — each one names the exact condition and the expected behavior. "What if something goes wrong" is not an edge case.

6. **Analytics events are named properly** — `collaborator_invite_accepted` tells you who, what, and when. `invite_success` tells you nothing.

7. **Eng tasks have one acceptance criterion each** — tasks with two acceptance criteria are two tasks.

8. **"What to do next" is actionable** — it doesn't say "next steps are to refine the spec." It says exactly what decision must be made, by whom, and why it's the riskiest assumption.
